What the Decision Intelligence Workflow Does

This workflow handles a compound analytical task that is usually rebuilt from scratch each time an executive asks why gross margin changed. It pulls transaction and cost history across comparison periods, calculates gross margin movement, ranks the contributing SKU-level signals, and packages the result into a briefing format a director can review quickly.

Click any step below to see the business logic, data query, and sample output for that step of the workflow.

Pre-specified logic, not runtime guessing — Most AI agent frameworks work by figuring things out on the fly. These Decision Intelligence workflows work differently. The Knowledge Work Foundry analyzes the cognitive labor pattern before deployment and encodes the decision logic directly into the configuration — which tables to query, which thresholds define a breach, how signals are ranked, and what the output artifact should contain. That analysis happens once. By the time the workflow runs, there is nothing left to figure out.

Under the Hood

            
Data Warehouse Integration

            
The workflow depends on clean access to both commercial and cost data, because gross margin explanations break down when revenue-side signals are separated from COGS history. It joins sales order lines with effective-dated SKU cost records so current and comparison periods are measured on the same basis.

            
That cross-domain integration matters because discount intensity, product mix, and unit cost shifts can each produce similar headline margin movement while requiring different management actions.

                    Source system: Sales transaction system output  · 
                    Domain: Sales Analytics

                    Role in this workflow: This table provides the transactional sales record needed to compare current and baseline periods by region, channel, SKU, and product line. It carries the units, list price, net price, and discount fields that allow the workflow to calculate realized revenue, discount intensity, and mix changes. It is the primary source for identifying whether margin movement is tied to pricing behavior or product mix.
                  

                

                
CREATE TABLE foods_dev.sales_dw.fact_sales_order_lines (
  order_id STRING,
  order_date DATE,
  region_id STRING,
  channel STRING,
  sku_id STRING,
  product_line STRING,
  units INT,
  list_price DECIMAL(10,2),
  net_price DECIMAL(10,2),
  discount_amount DECIMAL(10,2)
);

SELECT
  period_label,
  SUM(units) AS units,
  SUM(units * net_price) AS revenue,
  SUM(units * unit_cogs) AS cogs,
  (SUM(units * net_price) - SUM(units * unit_cogs)) / NULLIF(SUM(units * net_price), 0) AS gross_margin_pct,
  AVG((list_price - net_price) / NULLIF(list_price, 0)) AS avg_discount_rate
FROM (
  SELECT
    CASE
      WHEN s.order_date BETWEEN string">'{start_date}' AND string">'{end_date}' THEN string">'current'
      WHEN s.order_date BETWEEN string">'{compare_start_date}' AND string">'{compare_end_date}' THEN string">'compare'
    END AS period_label,
    s.units,
    s.list_price,
    s.net_price,
    c.unit_cogs
  FROM foods_dev.sales_dw.fact_sales_order_lines s
  JOIN foods_dev.sales_dw.dim_sku_cost c
    ON s.sku_id = c.sku_id
   AND s.order_date BETWEEN c.effective_start_date AND c.effective_end_date
  WHERE s.region_id = string">'{region_id}'
    AND s.channel = string">'{channel}'
    AND (
      s.order_date BETWEEN string">'{start_date}' AND string">'{end_date}'
      OR s.order_date BETWEEN string">'{compare_start_date}' AND string">'{compare_end_date}'
    )
) t
GROUP BY period_label
ORDER BY period_label;

              

              

                

                  

                    Source system: Cost and product master history  · 
                    Domain: Finance

                    Role in this workflow: This table supplies effective-dated unit COGS by SKU so each sale can be matched to the correct cost at the time of the order. It is what lets the workflow distinguish a revenue-side margin issue from a cost-side issue. When gross profit falls without a matching discount signal, this table is the source used to test whether unit cost increased.

Cognitive Labor Analysis

              
Where the Work Sits in the Labor Stack

              
Not all cognitive labor is equally automatable. The KWF analysis breaks the workflow into three layers — execution, judgment, and strategic — and maps each step to the layer it belongs to. Execution-layer work is automatable. Judgment and strategic work stays with the manager.

              
              

                

                

                

              

              

                 Execution 60%
                 Judgment 40%
                 Strategic 0%
              

              

                

                  

                    
Execution Layer

                    
60%

                    
The execution layer retrieves sales and cost data, reconciles effective-dated COGS, computes discount and margin metrics, ranks driver signals, and formats the evidence package.

                  

                

                

                  

                    
Judgment Layer

                    
40%

                    
The analyst still interprets whether the ranked signals are plausible in context, decides which actions to recommend, and handles leadership follow-up questions.

                  

                

                

                  

                    
Strategic Layer

                    
0%

                    
Strategic choices such as pricing policy changes, promo governance changes, supplier strategy, and assortment decisions remain with business leadership.
