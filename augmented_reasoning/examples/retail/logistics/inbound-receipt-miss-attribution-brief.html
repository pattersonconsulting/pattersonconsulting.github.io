How It Works

What the Decision Intelligence Workflow Does

This workflow handles a compound operations task that usually lives in spreadsheets, dashboards, and email threads. It pulls promised order details, shipment notice timing and quantities, and actual receipt outcomes into one analysis, then classifies miss types, attributes likely causes, and assembles a usable operating brief. The hard part is not one calculation; it is making the causal story consistent enough for procurement and receiving to act on.

Click any step below to see the business logic, data query, and sample output for that step of the workflow.

Pre-specified logic, not runtime guessing — Most AI agent frameworks work by figuring things out on the fly. These Decision Intelligence workflows work differently. The Knowledge Work Foundry analyzes the cognitive labor pattern before deployment and encodes the decision logic directly into the configuration — which tables to query, which thresholds define a breach, how signals are ranked, and what the output artifact should contain. That analysis happens once. By the time the workflow runs, there is nothing left to figure out.

The output is a one-page inbound exception attribution brief that summarizes miss rates, ranked drivers, supporting evidence, and recommended actions. It can be used in monthly vendor reviews, exception escalations, and receiving plan adjustments.

              

                What this workflow does NOT do: It does not decide whether to escalate a supplier into formal contract or chargeback review, determine whether warehouse staffing schedules should be changed, choose which supplier issue should take priority over other inbound risks, approve changes to vendor ASN policy, or renegotiate lane strategy or supplier terms.
              

            

          

        

      


      
      

        

          

            
Under the Hood

            
Data Warehouse Integration

            
The workflow depends on joining operational facts that usually sit in separate domains and are reviewed by different teams. It needs purchase order promise data, shipment notice timing and quantities, and warehouse receipt outcomes to determine whether a miss is more likely tied to supplier process, fulfillment, quality, or internal receiving execution.

            
That cross-domain join is the core execution burden today: procurement data, EDI shipment data, and warehouse receipt data have to be reconciled at PO-line level before anyone can explain the miss with confidence.

                    Source system: ERP purchase order records  · 
                    Domain: Procurement

                    Role in this workflow: This table defines the expected inbound commitment: supplier, region, PO line, promised delivery date, and ordered quantity. It is the baseline for determining whether a receipt was late, short, or missing, and it anchors the analysis window for the manager's review.
                  

                

                
CREATE TABLE pura_vida_foods_dev.inbound_ops.erp_po_lines (
  po_id STRING,
  po_line_id STRING,
  supplier_id STRING,
  region_id STRING,
  sku_id STRING,
  promised_delivery_date DATE,
  ordered_qty INT,
  unit_cost_usd DECIMAL(10,2),
  incoterm STRING,
  created_ts TIMESTAMP
);



              

              

                

                  

                    Source system: Warehouse Management System (WMS) output  · 
                    Domain: Warehouse Operations

                    Role in this workflow: This table shows what the warehouse actually received, when it was received, in what quantity, and whether the line was placed on QC hold. It provides the operational truth used to classify missed receipts and detect internal receiving or quality-related delay patterns.
                  

                

                
CREATE TABLE pura_vida_foods_dev.inbound_ops.wms_receipts (
  receipt_id STRING,
  po_id STRING,
  po_line_id STRING,
  region_id STRING,
  received_date DATE,
  received_qty INT,
  qc_hold_flag BOOLEAN,
  created_ts TIMESTAMP
);

SELECT
  p.supplier_id,
  p.region_id,
  COUNT(*) AS po_lines,
  SUM(CASE WHEN r.receipt_id IS NULL THEN 1 ELSE 0 END) AS not_received_lines,
  SUM(CASE WHEN r.receipt_id IS NOT NULL AND r.received_date > p.promised_delivery_date THEN 1 ELSE 0 END) AS late_lines,
  SUM(CASE WHEN r.receipt_id IS NOT NULL AND r.received_qty < p.ordered_qty THEN 1 ELSE 0 END) AS short_lines,
  SUM(CASE WHEN r.receipt_id IS NOT NULL AND r.qc_hold_flag = true THEN 1 ELSE 0 END) AS qc_hold_lines
FROM pura_vida_foods_dev.inbound_ops.erp_po_lines p
LEFT JOIN pura_vida_foods_dev.inbound_ops.wms_receipts r
  ON p.po_id = r.po_id AND p.po_line_id = r.po_line_id AND p.region_id = r.region_id
WHERE p.supplier_id = string">'{supplier_id}'
  AND p.region_id = string">'{region_id}'
  AND p.promised_delivery_date BETWEEN string">'{start_date}' AND string">'{end_date}'
GROUP BY p.supplier_id, p.region_id;

Cognitive Labor Analysis

              
Where the Work Sits in the Labor Stack

              
Not all cognitive labor is equally automatable. The KWF analysis breaks the workflow into three layers — execution, judgment, and strategic — and maps each step to the layer it belongs to. Execution-layer work is automatable. Judgment and strategic work stays with the manager.

              
              

                

                

                

              

              

                 Execution 70%
                 Judgment 25%
                 Strategic 5%
              

              

                

                  

                    
Execution Layer

                    
70%

                    
The execution layer retrieves PO, ASN, and receipt data, reconciles exceptions, calculates contribution shares, evaluates thresholds, and assembles the briefing narrative.

                  

                

                

                  

                    
Judgment Layer

                    
25%

                    
The manager still interprets the evidence in context, decides whether the issue is supplier-side or internal, and chooses the corrective action path.

                  

                

                

                  

                    
Strategic Layer

                    
5%

                    
Strategic decisions such as supplier governance changes, penalty policy, and lane reliability programs remain with leadership.
