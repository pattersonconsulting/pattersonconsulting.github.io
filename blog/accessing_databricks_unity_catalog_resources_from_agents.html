Accessing Unity Catalog Resources from Databricks Framework Agents

By: Nikhil Karnik and Josh Patterson

A Databricks agent never just “sprays SQL at Unity Catalog.” It reaches Unity Catalog tables and metric views through tools that are implemented as Unity Catalog functions whose bodies run SQL (plus, optionally, Genie for free-form SQL). Here’s how that actually works end-to-end.

1. The pattern: SQL lives inside Unity Catalog functions, not in the prompt

The structured-retrieval docs call out two main ways to let an agent query structured data:

  1. Unity Catalog SQL functions as tools
    You define a SQL (or Python) function in Unity Catalog. The function body contains the SELECT against your tables or metric views. The function’s signature and docstring/COMMENT become the tool description the LLM sees. The agent fills in the parameters; Databricks runs the function on serverless compute.
  2. Genie multi-agent systems (SQL generation)
    You add a Genie agent that’s allowed to see certain catalogs/schemas. Genie reasons in natural language and auto-generates SQL over those tables/metric views.

In both cases, the LLM is choosing and parameterizing tools, not writing raw SQL inline.

2. Accessing Unity Catalog tables via SQL functions

Step 1: Create a UC function whose body runs SQL

From the docs example, you create a function that selects from a UC table: 


CREATE OR REPLACE FUNCTION main.default.lookup_customer_info(
  customer_name STRING COMMENT 'Name of the customer whose info to look up'
)
RETURNS STRING
COMMENT 'Returns metadata about a particular customer, including email and ID.'
RETURN SELECT CONCAT(
    'Customer ID: ', customer_id, ', ',
    'Customer Email: ', customer_email
  )
  FROM main.default.customer_data
  WHERE customer_name = customer_name
  LIMIT 1;

Key points:

  • The SQL body (SELECT ... FROM main.default.customer_data) is what actually hits the Unity Catalog table.
  • The function is fully qualified in UC (main.default.lookup_customer_info), so governance, lineage, permissions, etc. all apply.
  • SQL here can aggregate, join, or shape results specifically for the LLM.

Step 2: Wrap the UC function as an agent tool


from unitycatalog.ai.core.databricks import DatabricksFunctionClient
from databricks_langchain import UCFunctionToolkit

client = DatabricksFunctionClient()

CATALOG = "main"
SCHEMA = "default"

# (Assume lookup_customer_info already created via SQL or Python)

func_name = f"{CATALOG}.{SCHEMA}.lookup_customer_info"
toolkit = UCFunctionToolkit(function_names=[func_name])

tools = toolkit.tools  # these are the tools the LLM sees

UCFunctionToolkit:

  • Reads the function metadata & comments.
  • Builds a tool schema that matches tool-calling LLM expectations.
  • Adds optional tracing and instrumentation.

Step 3: Plug tools into your agent

Then you pass tools into your agent framework (LangChain, MLflow ResponsesAgent, OpenAI/Anthropic tool-calling, etc.).

At runtime:

  1. User: "What is the email for customer Jane Doe?"
  2. LLM sees tool lookup_customer_info(customer_name: string).
  3. LLM emits a tool call: 
    {
  "tool": "lookup_customer_info",
  "arguments": {"customer_name": "Jane Doe"}
}
  4. Databricks executes the UC function server-side.
  5. Function runs SQL and returns a value.
  6. LLM responds using the result.

3. Accessing Unity Catalog metric views via SQL

Metric views are first-class SQL objects that define reusable metrics on top of tables. They’re queried with SQL, but measures must be referenced via MEASURE() and SELECT * is not allowed.

Example UC function that queries a metric view: 


CREATE OR REPLACE FUNCTION retail_analytics.tools.get_otif_for_supplier(
  supplier_id STRING COMMENT 'The supplier to analyze',
  start_date  DATE   COMMENT 'Start date (inclusive)',
  end_date    DATE   COMMENT 'End date (inclusive)'
)
RETURNS STRING
COMMENT 'Returns daily OTIF percentages for a supplier over a date range, as CSV.'
RETURN
  SELECT to_csv(named_struct(
           'order_date', order_date,
           'otif_rate',  MEASURE(otif_rate)
         ))
  FROM retail_analytics.gold.mv_otif
  WHERE supplier_id = supplier_id
    AND order_date BETWEEN start_date AND end_date;

This enables questions like: "What was OTIF for Supplier X last quarter?"

4. Governance & execution path (why this matters)

Because these tools are UC functions:

  • Unity Catalog permissions and data policies apply automatically.
  • Execution runs on Databricks serverless compute.
  • The LLM sees only function inputs and outputs, limiting data-leak risk.
  • Auditing & lineage track data usage.

For metric views you also inherit:

  • Consistent KPI definitions across BI, SQL, and AI.
  • Centralized YAML metric logic.
  • No need to modify agent code when metric logic changes.

5. Where Genie fits

If you don’t want to predefine every function, you can use Genie multi-agent systems:

  • Grant Genie access to catalogs/schemas.
  • Genie inspects schema metadata.
  • Generates SQL and retrieves results.

So:
UC functions as tools → curated, governed workflows
Genie → ad-hoc flexible SQL reasoning

Quick mental model

Databricks agent → UC function tools → SQL → Unity Catalog tables/metric views

You put the SQL into Unity Catalog functions. You wrap those functions as tools. The agent calls tools, not raw SQL. The SQL runs against governed Unity Catalog data.
