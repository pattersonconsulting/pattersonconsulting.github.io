Why Build Agents in the First Place?

The objective of agentic systems is not simply to be ‚Äúagentic‚Äù‚Äîthe real goal is accelerating the transformation of information into decisions. Organizations improve through only three mechanisms: automating work, improving information flow, and improving decision-making. Agents live directly next to enterprise data, enhancing both information flow and decision quality.

Large language models (LLMs) are rapidly evolving from single-prompt utilities into full agentic systems capable of using tools, retrieving structured information, managing memory, and performing multi-step reasoning. But building these agents at an enterprise level remains difficult. To address this challenge, organizations are increasingly turning to MLflow as the runtime for production-grade generative systems.

When agents run inside a governed platform like Unity Catalog, their access to data, tools, and execution context becomes safer and more reliable. This is where MLflow becomes essential: it provides consistent packaging, lineage, permissions, and deployment guarantees that enterprises require before allowing agents to operate on sensitive data.

Why Building Agents Is Hard

Building production agents introduces challenges across three dimensions: quality, control, and cost. First, high quality is difficult to enforce‚ÄîLLMs can hallucinate, drift in behavior, or generate inconsistent structured outputs. Many enterprise workflows require domain experts to validate and shape behavior, meaning evaluation must go beyond prompt engineering.

Second, enterprises cannot operate agents without strong governance. Without proper guardrails, agents risk leaking sensitive data, violating compliance rules, or producing opaque reasoning paths. Compliance frameworks like HIPAA and SOC2 require traceability, permissions, and visibility into reasoning steps, tool calls, and lineage. Finally, cost management is unavoidable: high-performing models are expensive, tool calls add latency, and the lack of centralized monitoring can lead to runaway spend. MLflow directly addresses these issues with standardized packaging, tracing, evaluation, and deployment infrastructure.

Why MLflow Is the Right Abstraction for Agents

MLflow provides the four core pillars of enterprise ML‚ÄîTracking, Models, Registry, and Serving. These abstractions standardize how code, environments, metrics, and deployments move through a pipeline. Agents fit naturally into this system because MLflow now treats them as first-class generative models with typed signatures.

This makes agents operationally simple: they inherit the same lifecycle as any other MLflow model. They can be versioned, governed, permissioned, traced, and deployed to Mosaic AI Model Serving in a single command. They integrate seamlessly with AI Playground, agent evaluation workflows, and MLflow lineage, tying together code ‚Üí data ‚Üí outputs ‚Üí deployed behavior. Since MLflow is already embedded in most enterprise MLOps stacks, adopting agents becomes almost frictionless.

Introducing ResponsesAgent

ResponsesAgent is MLflow‚Äôs purpose-built model interface for building generative and tool-using agents. It standardizes conversational output, tool-calling semantics, long-running tool operations, streaming, tracing, and automatic model signature inference. With ResponsesAgent, developers write clean Python classes that encapsulate agent behavior while MLflow handles packaging, reproducibility, deployment, and governance. The ResponsesAgent in MLflow 3.6 is a specialized model interface designed for modern conversational AI applications. It extends the traditional PyFunc abstraction to support multi-turn dialogue, tool-calling workflows, agent-to-agent communication, and structured OpenAI-style requests and responses. This makes it a strong foundation for chatbots, decision assistants, and automated knowledge-work applications needing governance, traceability, and repeatable behavior.

Because the ResponsesAgent is fully integrated into MLflow‚Äôs lifecycle ‚Äî including tracking, lineage, registry, and serving ‚Äî teams can deploy advanced generative AI solutions with the same operational rigor used for traditional ML models. It enables rapid prototyping while supporting safe and scalable production deployments.

Building a Simple ResponsesAgent

Below is a minimal example of an agent that uses tool-calling to perform math operations. This example demonstrates the core structure of a ResponsesAgent and how MLflow manages typing and signatures. 


from mlflow import MlflowClient
from mlflow.agents import ResponsesAgent
from openai import OpenAI

client = OpenAI()   # or DatabricksOpenAI()

class MathAgent(ResponsesAgent):

    def __init__(self):
        super().__init__()

    def predict(self, messages, tools):
        """
        Basic agent with tool-calling.
        """
        response = client.chat.completions.create(
            model="gpt-4o-mini",
            messages=messages,
            tools=tools,
            tool_choice="auto"
        )
        return response

MLflow automatically infers the typed signature: messages ‚Üí messages. No manual schema definition is required.

The ResponsesAgent introduces a powerful and standardized approach to orchestrating LLM-driven workflows with structured inputs and outputs.

OpenAI's Chat Completions API

OpenAI‚Äôs Chat Completions API is a hosted service that lets applications send a structured list of messages‚Äîsystem, user, and assistant‚Äîto a large language model and receive a generated response. It is the API behind most modern conversational and reasoning-style interactions with OpenAI models. Developers use it to run instruction following, multi-turn dialogues, tool-calling, and structured output generation. In short, it is the primary interface for invoking OpenAI models to produce conversational, analytical, or task-oriented responses in applications.
