What is MLflow?

By: Patterson Consulting Engineering Team

This article is focused on what is MLflow and how it can help you deploy and integrate agents into applications.

what IS mlflow?

MLflow provides a unified framework to manage the entire lifecycle of both traditional ML models and modern LLM agents — from experiment tracking and versioning to packaging, deployment, and governance. It ensures every step is reproducible, auditable, and easy to operationalize, so teams can iterate quickly during development and still maintain high standards of reliability and control in production.

A Framework to Support Agent Application Architectures

Running agents on MLflow standardizes how teams build, package, and operate agentic systems. Every agent follows the same interface and deployment lifecycle, dramatically improving operational consistency and reducing friction across teams. Unity Catalog adds enterprise-grade governance, ensuring agent code, tool access, data permissions, and deployment states are all fully auditable.

Beyond governance, MLflow delivers deep observability and repeatability. MLflow Tracing captures messages, tool calls, intermediate reasoning, latency, and cost—giving teams end-to-end visibility into behavior. With versioning, shadow deployments, rollbacks, and side-by-side evaluations, MLflow allows rapid iteration while maintaining production safety and stability.

... 4 pillars

MLflow provides the four core pillars of enterprise ML—Tracking, Models, Registry, and Serving. These abstractions standardize how code, environments, metrics, and deployments move through a pipeline. Agents fit naturally into this system because MLflow now treats them as first-class generative models with typed signatures.

This makes agents operationally simple: they inherit the same lifecycle as any other MLflow model. They can be versioned, governed, permissioned, traced, and deployed to Mosaic AI Model Serving in a single command. They integrate seamlessly with AI Playground, agent evaluation workflows, and MLflow lineage, tying together code → data → outputs → deployed behavior. Since MLflow is already embedded in most enterprise MLOps stacks, adopting agents becomes almost frictionless.

MLflow is organized around four core pillars that support the end-to-end machine learning lifecycle — plus a rapidly growing fifth emerging component focused on real-time production impact.

Logging a Model on MLflow

It’s important to distinguish between logging a model and registering a model:

  • Logging a Model: Saves a model artifact to the MLflow Tracking Server as part of an experiment run.
  • Registering a Model: Adds the saved model to the centralized Model Registry for lifecycle governance and deployment workflows.

Together, these capabilities provide a robust foundation for promoting stable, traceable models into production environments.

Model Registry

...

The Model Registry is the system of record for all production models. It manages versioning, approvals, stages (Staging → Production), and governance. This ensures the right model is always running with full auditability and lineage.

The MLflow Model Registry provides a centralized system for managing all production-bound machine learning models with structure, governance, and collaboration in mind.

  • Centralized Model Store: Acts as the single source of truth for all MLflow models — simplifying versioning, discovery, and deployment across teams and environments.
  • Rich APIs: Enables programmatic creation, retrieval, updating, and archiving of models, empowering automation across the entire MLOps lifecycle.
  • User Interface: A visual dashboard that allows practitioners to inspect versions, transitions, lineage, and model ownership directly from the MLflow UI.

With these capabilities, the Model Registry ensures models move from experimentation to production with traceability, confidence, and control.

Key Benefits of the Model Registry

  • Centralized Store: A single place to manage all your models.
  • Versioning: Tracks new versions automatically (e.g., Model-1, Model-2).
  • Lifecycle Stages: Transition models to Staging, Production, or Archived.
  • Discovery: Helps teams find and use models across projects.

Options for Running the Model Registry

The MLflow Model Registry is supported in both open-source (OSS) MLflow and managed enterprise platforms such as Databricks. The functionality remains consistent, but managed platforms extend governance, collaboration, and security capabilities for production-scale operations.

In the OSS version of MLflow, users can manage the full lifecycle of machine learning models through both a UI and rich APIs. This includes registering models, tracking version history, applying tags and descriptions, and promoting models between stages such as Staging and Production.

Deployments (Emerging)

The newest area of expansion in MLflow is integrated Deployments / Model Serving. This simplifies the operational footprint required to deliver real-time inference — directly from the MLflow platform — while capturing cost, telemetry, and performance data for continuous improvement.

Tracking

MLflow Tracking allows teams to log, compare, and visualize all the artifacts that matter.

An artifact is any file or directory that you log alongside a run or a model—such as data files, configuration files, prompts, source code, images, or serialized model assets—so that MLflow can store, version, and later retrieve them as part of the model’s reproducible package.

Artifacts are not parameters or metrics; they are actual files that MLflow saves in the run’s artifact store or inside an MLflow model’s artifact directory.

Parameters, metrics, model versions, datasets, and other outputs are easily compared — enabling highly collaborative and reproducible experimentation.

Tracking an Agent

MLflow Tracking and ResponsesAgent work together closely in MLflow 3.6, but each serves a different role. Tracking and Tracing act as the “flight recorder,” while the agent itself — your logic, tools, and model calls — is the “thing that flies.” When you run an agent packaged and deployed through MLflow, every prediction call can generate a trace of the conversation, tool calls, and intermediate steps, which is automatically captured by MLflow Tracking if a tracking server is configured.

The interaction happens across three layers. First, the ResponsesAgent defines the system instructions, tool usage, and behavior routing. Second, you package and deploy the agent as an MLflow Model using mlflow.agents.log_model, which wraps your agent in a standardized predict() interface. Finally, when that model is run — locally via mlflow.pyfunc.load_model() or through a model serving endpoint — MLflow Tracking and Tracing automatically record the entire execution: inputs, outputs, intermediate messages, performance metrics, and any custom logging you add. In the MLflow UI, this information appears both as traditional experiment runs and in the “Traces” view for step-by-step agent behavior.

This means you gain observability, governance, repeatability, and lineage for agents just as you would for more traditional ML models. Agents evolve rapidly, so capturing these details is essential for debugging and comparing versions. By logging the agent as a model and running it through MLflow’s interfaces, you get tracking “for free,” helping you understand how prompts perform, which tools get invoked, and how costs and latency behave. If running in Databricks, Unity Catalog further provides access controls and lineage over data and tools your agent uses.

1. Where Tracking Shows Up With ResponsesAgent

When you run an agent that’s packaged and deployed via MLflow 3.6:

– The agent call itself is treated as an MLflow “generation” (or series of generations). – Each call can generate:

  • A trace: tree/graph of messages, tool calls, sub-agents, etc.
  • One or more runs / logged artifacts attached to that trace.

In other words:

You call agent.predict() (or hit the model’s REST endpoint) → MLflow Tracking automatically records the conversation, parameters, tools used, and outputs as structured telemetry.

Depending on how you configure it, this shows up in:

  • The MLflow UI “Traces” view (for step-by-step agent behavior).
  • The usual runs/artifacts view (for parameters, metrics, and artifacts you explicitly log yourself).

The tracking server is responsible for recording and organizing experiments — including runs, parameters, metrics, artifacts, and model versions. It acts as the central system of record for experimentation and model lifecycle history.

The model server, on the other hand, is where trained or authored models are deployed and made available for real-time or batch inference. It provides a standardized API to call a model’s predict() function.

These two components operate independently:

  • You can serve a model locally without ever running an MLflow tracking server.
  • You can use a tracking server to log experiments and models without deploying any model for inference.

Key Concepts

Runs

A run is the atomic tracked execution in MLflow—it represents a single attempt to create or update a model or agent. But its purpose differs slightly depending on the asset type.

The conceptual difference between artifacts and runs in MLflow 3.6:

A run is the container for a single execution of something you are tracking in MLflow.

Think of a run as the record or envelope that captures:

– parameters used
– metrics produced
– tags
– the execution context
– and any artifacts logged during that execution

A run is essentially the metadata log for one experiment attempt.

If you did an experiment, executed a script, trained a model, or invoked an agent, that execution is represented as a run.

In the context of deploying and calling the predict method on an MLflow 3.6 ResponsesAgent, a Run is the formal MLflow tracking record that represents a single execution of the agent’s logic. It captures the input, output, metadata, and any artifacts produced during that specific invocation.

What a Run Represents

A Run is created when the agent’s predict() method is called (assuming logging is enabled). This run becomes the container for all information related to that individual inference event. It does not represent deployment, model registration, or serving infrastructure changes—only the execution itself.

What a Run Records

During a predict() call, a run may contain:

– the user input messages
– the agent’s generated outputs
– reasoning traces (if enabled)
– tool call logs
– metrics or timing information
– any artifacts produced during the interaction

How Runs Relate to Predict Calls

Each call to predict() is treated as one run because MLflow considers each invocation of the agent to be an independently loggable execution. Runs provide the audit trail, observability, and reproducibility for every inference.

Summary: In MLflow 3.6, a Run represents a single, logged execution of a ResponsesAgent.predict() call, capturing everything that happened during that inference event.

Artifact

Artifacts are files produced by a run (models, plots, metrics, configs).

MLflow Artifacts are any files (models, plots, data, configs) generated during an ML experiment run, stored separately from metadata, accessible via URIs (like runs://path), and crucial for reproducibility, allowing you to log, retrieve, and manage outputs like model files (pickle, PyTorch), visualizations (plots), and preprocessing data, with storage configured locally (mlruns) or remotely (S3, Databricks)

An artifact is any file or directory that you log alongside a run or a model—such as data files, configuration files, prompts, source code, images, or serialized model assets—so that MLflow can store, version, and later retrieve them as part of the model’s reproducible package.

Artifacts are not parameters or metrics; they are actual files that MLflow saves in the run’s artifact store or inside an MLflow model’s artifact directory.

Artifacts are the files produced during a run.

Examples:

– model files
– prompt templates
– Python source code snapshots
– plots, images
– datasets
– configuration files

Artifacts are outputs stored in MLflow’s artifact store. A run owns artifacts, and MLflow tracks exactly which artifacts were generated by which run.

Think of artifacts as the stuff you produced, and a run as the record of the attempt that produced them.

Relating Runs and Artifacts

– Run = a folder labeled with the metadata about an experiment attempt. (where a run is a call of the .predict() ResponsesAgent method)
– Artifacts = the files inside that folder.

Traces

In the context of deploying and calling the predict method on an MLflow 3.6 ResponsesAgent, a Trace is the detailed, structured record of the internal steps the agent takes while generating a response. It captures the reasoning process, tool interactions, intermediate states, and decision paths executed during a single prediction call.

What a Trace Represents

A trace is not the same as a run. A run represents the overall execution event, whereas a trace represents the internal workflow the agent follows within that event. The trace provides visibility into how the agent arrived at its final answer.

What a Trace Records

A trace may include:

– the sequence of steps taken by the agent
– calls to tools or external functions
– intermediate messages and decisions
– reasoning metadata (when enabled)
– timing information for each step
– outputs generated at each stage of the process

The trace essentially exposes the chain of thought structure without revealing any private or disallowed reasoning content.

How Traces Relate to Predict Calls

Each call to predict() may generate a trace if tracing is enabled. The trace becomes part of the run’s logged artifacts, making it possible to debug, audit, or analyze how the agent behaved during execution.

Summary: In MLflow 3.6, a Trace is the internal step-by-step execution record of a ResponsesAgent.predict() call, capturing the sequence of tool calls, intermediate decisions, and reasoning metadata that led to the final output.
